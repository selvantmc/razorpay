# Requirements Document

## Introduction

The Nourisha POS Payment Module is a secure, reusable Flutter component for processing payments through Razorpay in Point-of-Sale environments. This module is designed for food ordering and billing counters, prioritizing security, reliability, and payment reconciliation capabilities. The module will be used internally by Nourisha and made available to third-party developers integrating with Nourisha POS systems.

## Glossary

- **Payment_Module**: The Flutter component that handles payment processing through Razorpay
- **POS_Screen**: The user interface component for payment initiation and status display
- **Payment_Service**: The service layer that manages Razorpay SDK integration and payment flow
- **Backend_API**: Server-side endpoints that handle secure payment operations (order creation, verification, status checks)
- **Razorpay_Checkout**: The Razorpay SDK payment interface launched within the Flutter app
- **Order_ID**: Unique identifier for a payment order generated by the backend
- **Payment_ID**: Unique identifier for a completed payment transaction from Razorpay
- **Payment_Status**: The state of a payment (success, failed, pending)
- **Key_ID**: Razorpay public key identifier (safe for client-side use)
- **Key_Secret**: Razorpay private key (MUST only exist on backend)
- **Payment_Callback**: Event handler triggered by Razorpay after payment attempt
- **Reconciliation**: Process of verifying payment status through backend API when callback is lost

## Requirements

### Requirement 1: Security Architecture

**User Story:** As a security auditor, I want payment credentials properly segregated, so that the application is production-safe and compliant with payment security standards.

#### Acceptance Criteria

1. THE Payment_Module SHALL NOT contain any Razorpay Key_Secret values
2. THE Payment_Module SHALL use only Razorpay Key_ID for client-side operations
3. WHEN order creation is requested, THE Payment_Module SHALL call Backend_API endpoints
4. WHEN payment verification is needed, THE Payment_Module SHALL call Backend_API endpoints
5. WHEN payment status checks are performed, THE Payment_Module SHALL call Backend_API endpoints
6. THE Payment_Module SHALL include inline comments documenting security boundaries between client and server

### Requirement 2: Payment Initiation Interface

**User Story:** As a cashier, I want a clear payment screen with amount entry, so that I can quickly process customer payments at the counter.

#### Acceptance Criteria

1. THE POS_Screen SHALL display the title "Nourisha POS – Payment"
2. THE POS_Screen SHALL provide an amount input field accepting numeric values with rupee symbol
3. THE POS_Screen SHALL render the amount input field in large font for visibility
4. THE POS_Screen SHALL provide an optional order reference field for notes
5. THE POS_Screen SHALL display a "Pay Now" button with large touch target
6. THE POS_Screen SHALL use minimal colors suitable for POS environments
7. THE POS_Screen SHALL use touch targets sized for counter operation

### Requirement 3: Razorpay Payment Flow

**User Story:** As a developer, I want the payment flow to follow Razorpay best practices, so that payments are processed reliably and securely.

#### Acceptance Criteria

1. WHEN the "Pay Now" button is pressed, THE Payment_Service SHALL call Backend_API to create an order
2. WHEN the Backend_API returns an order response, THE Payment_Service SHALL extract the Order_ID
3. WHEN an Order_ID is received, THE Payment_Service SHALL launch Razorpay_Checkout with the Order_ID, amount, and Key_ID
4. THE Payment_Service SHALL configure Razorpay_Checkout with Nourisha branding information
5. THE Payment_Service SHALL register callback handlers for onPaymentSuccess, onPaymentError, and onExternalWallet events
6. WHEN onPaymentSuccess is triggered, THE Payment_Service SHALL capture the Payment_ID and signature
7. WHEN onPaymentError is triggered, THE Payment_Service SHALL capture the error code and description
8. WHEN onExternalWallet is triggered, THE Payment_Service SHALL capture the wallet name

### Requirement 4: Response Display and State Management

**User Story:** As a cashier, I want to see payment results clearly displayed, so that I can confirm transaction status to customers.

#### Acceptance Criteria

1. THE POS_Screen SHALL display a scrollable response panel for payment results
2. WHEN a payment response is received, THE POS_Screen SHALL format the response as structured JSON
3. THE POS_Screen SHALL store the most recent Order_ID in application state
4. THE POS_Screen SHALL store the most recent Payment_ID in application state
5. THE POS_Screen SHALL store the most recent Payment_Status in application state
6. WHEN payment succeeds, THE POS_Screen SHALL display status in green color
7. WHEN payment fails, THE POS_Screen SHALL display status in red color
8. WHEN payment is pending, THE POS_Screen SHALL display status in orange color

### Requirement 5: Payment Failure and Edge Case Handling

**User Story:** As a cashier, I want the system to handle payment interruptions gracefully, so that I can recover from network issues or app crashes.

#### Acceptance Criteria

1. WHEN a Payment_Callback is not received within expected timeframe, THE Payment_Module SHALL maintain the last known Order_ID for reconciliation
2. WHEN the app is closed during payment, THE Payment_Module SHALL persist the Order_ID to local storage
3. WHEN the app is restarted after closure during payment, THE Payment_Module SHALL restore the last Order_ID
4. WHEN network failure occurs after payment submission, THE Payment_Module SHALL display a clear error message with reconciliation instructions
5. WHEN Backend_API calls fail, THE Payment_Module SHALL display the error response in the response panel

### Requirement 6: Payment Status Reconciliation

**User Story:** As a cashier, I want to manually check payment status, so that I can verify transactions when automatic confirmation fails.

#### Acceptance Criteria

1. THE POS_Screen SHALL display a "Check Payment Status" button
2. WHEN the "Check Payment Status" button is pressed, THE Payment_Service SHALL call Backend_API with the stored Order_ID or Payment_ID
3. WHEN the Backend_API returns payment status, THE Payment_Service SHALL parse the response into success, failed, or pending states
4. WHEN payment status is retrieved, THE POS_Screen SHALL update the response panel with the current status
5. THE Payment_Service SHALL support status checks even when initial Payment_Callback was not received

### Requirement 7: Code Architecture and Modularity

**User Story:** As a developer, I want clean separation of concerns, so that the payment module is maintainable and reusable across projects.

#### Acceptance Criteria

1. THE Payment_Module SHALL implement payment logic in a dedicated payment_service.dart file
2. THE Payment_Module SHALL implement UI components in a dedicated razorpay_pos_screen.dart file
3. THE Payment_Module SHALL define data structures in a dedicated payment_models.dart file
4. THE payment_models.dart file SHALL include OrderResponse, PaymentResult, and PaymentStatus models
5. THE Payment_Module SHALL implement Backend_API calls in a separate service with clear mock implementations
6. THE Payment_Module SHALL include TODO comments marking mock backend endpoints for production replacement
7. THE Payment_Service SHALL initialize Razorpay SDK with configuration parameters
8. THE Payment_Service SHALL expose an openCheckout method for launching payment flow

### Requirement 8: Backend API Integration

**User Story:** As a backend developer, I want clearly defined API contracts, so that I can implement secure server-side payment operations.

#### Acceptance Criteria

1. THE Payment_Module SHALL define a createOrder API endpoint contract accepting amount and optional reference
2. THE createOrder endpoint SHALL return Order_ID, amount, and currency in the response
3. THE Payment_Module SHALL define a verifyPayment API endpoint contract accepting Order_ID, Payment_ID, and signature
4. THE verifyPayment endpoint SHALL return verification status and payment details
5. THE Payment_Module SHALL define a checkPaymentStatus API endpoint contract accepting Order_ID or Payment_ID
6. THE checkPaymentStatus endpoint SHALL return current Payment_Status and transaction details
7. THE Payment_Module SHALL implement mock versions of all Backend_API endpoints with realistic response structures

### Requirement 9: Configuration and Deployment

**User Story:** As a developer, I want easy configuration for different environments, so that I can test with Razorpay test mode and deploy to production.

#### Acceptance Criteria

1. THE Payment_Module SHALL use a placeholder constant for Razorpay Test Key_ID
2. THE Payment_Module SHALL include comments indicating where to configure production Key_ID
3. THE Payment_Module SHALL use Razorpay test mode by default
4. THE Payment_Module SHALL include documentation for switching between test and production modes
5. THE Payment_Module SHALL validate that Key_Secret is never present in client code through code comments and architecture

### Requirement 10: Razorpay Flutter SDK Integration

**User Story:** As a developer, I want proper Razorpay SDK integration, so that the payment module uses official, maintained libraries.

#### Acceptance Criteria

1. THE Payment_Module SHALL depend on the razorpay_flutter package
2. THE Payment_Service SHALL initialize the Razorpay SDK instance on service creation
3. THE Payment_Service SHALL dispose of the Razorpay SDK instance when the service is destroyed
4. THE Payment_Service SHALL handle Razorpay SDK lifecycle events properly
5. THE Payment_Module SHALL include the razorpay_flutter dependency in pubspec.yaml with version constraints
